# Contributing to DeDi

Thanks for contributing.

This repository is still relatively early, so the best contributions are the ones that reduce ambiguity and improve implementability.

## High-value contribution areas

We especially welcome contributions that improve:

- developer onboarding,
- schema clarity and consistency,
- example payloads,
- protocol documentation,
- integration guidance,
- and ecosystem-specific mappings.

## Before you open a pull request

Please try to keep changes concrete and reviewable.

A strong contribution usually does one or more of the following:

1. clarifies a field or semantic ambiguity,
2. adds a realistic example,
3. improves guidance for implementers,
4. proposes a new schema with a clear use case,
5. or tightens consistency across the repository.

## Suggested workflow

1. Open an issue describing the problem, gap, or proposed change.
2. Explain the implementation impact, not only the conceptual idea.
3. Keep pull requests scoped where possible.
4. Update documentation and examples with the change.
5. Call out any backward-compatibility impact explicitly.

## Contribution checklist

Before submitting:

- [ ] Documentation is updated.
- [ ] Relevant examples are added or refreshed.
- [ ] Schema semantics are explained, not just changed.
- [ ] Backward compatibility impact is noted.
- [ ] File names and terms are consistent with the rest of the repo.

## Issue types that are especially useful

- unclear field definitions,
- missing discovery conventions,
- weak or absent examples,
- adoption blockers for real implementers,
- schema inconsistencies,
- protocol questions that need explicit guidance.

## Style guidance

Prefer:

- plain language,
- implementation-first wording,
- concrete examples,
- explicit semantics,
- and minimal ambiguity.

Avoid:

- product marketing language in protocol text,
- unexplained assumptions,
- and ecosystem-specific claims unless they are clearly labeled as examples.

## Compatibility and versioning

Treat schema changes carefully.

- Breaking field changes should be called out explicitly.
- New optional fields should explain their intended semantics.
- Example records should remain aligned with the current schema set.

## Where to start

A great first contribution is to add or improve:

- one realistic example payload,
- one missing schema explanation,
- or one piece of adoption guidance for developers.
